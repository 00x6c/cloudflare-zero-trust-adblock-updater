# Требования к дизайну интерфейса программы `cf-zt-oisd-sync`

## 1. Назначение документа

Этот документ описывает требования к пользовательскому интерфейсу программы `cf-zt-oisd-sync`, которая синхронизирует официальный список OISD small с Cloudflare Zero Trust Gateway.

Цель интерфейса — сделать работу с программой понятной для пользователя, который не является разработчиком, но умеет работать с Cloudflare, API token, Zero Trust и базовой настройкой DNS Gateway.

Интерфейс должен помогать пользователю безопасно:

1. подключить Cloudflare account;
2. проверить доступ к Cloudflare API;
3. скачать и проверить список OISD small;
4. создать списки и DNS rule;
5. обновлять списки;
6. удалять созданные программой объекты;
7. понимать, что именно происходит на каждом этапе.

---

## 2. Общая концепция интерфейса

Программа должна иметь CLI-интерфейс, но оформленный максимально дружелюбно и понятно.

CLI должен быть не «сухим скриптом», а полноценной консольной утилитой с:

- понятными командами;
- цветным выводом;
- таблицами;
- прогресс-барами;
- подтверждениями опасных действий;
- режимом предварительного просмотра изменений;
- человекочитаемыми ошибками;
- короткими подсказками, что делать дальше.

Основная идея интерфейса:

> Пользователь должен всегда понимать: что программа делает сейчас, что она собирается изменить в Cloudflare, что уже создано, что будет удалено и что делать при ошибке.

---

## 3. Целевая аудитория

Интерфейс рассчитан на три уровня пользователей.

### 3.1. Базовый пользователь

Умеет открыть терминал, скопировать команду, создать API token по инструкции, вставить значения в `.env`.

Для такого пользователя интерфейс должен быть максимально пошаговым.

### 3.2. Администратор Cloudflare Zero Trust

Понимает DNS Gateway, policies, reusable lists, API tokens, account ID.

Для него важны точность, статус, dry-run и отсутствие лишнего риска.

### 3.3. Разработчик/DevOps

Хочет автоматизировать запуск через Task Scheduler, cron или GitHub Actions.

Для него важны exit-коды, machine-readable режим и стабильные команды.

---

## 4. Основные принципы UX

### 4.1. Безопасность важнее скорости

Любые действия, которые создают, изменяют или удаляют объекты в Cloudflare, должны быть явно показаны пользователю до выполнения.

Удаление должно требовать подтверждения.

### 4.2. Dry-run поощряется

При первом запуске программа должна рекомендовать выполнить `dry-run`.

Например:

```text
Совет: перед первым применением запустите:
cf-zt-oisd-sync dry-run
```

### 4.3. Не заставлять пользователя читать сырые JSON-ответы Cloudflare

Ошибки Cloudflare API нужно переводить в понятный вид:

```text
Ошибка Cloudflare API: недостаточно прав токена.
Проверьте, что API token имеет права на Zero Trust Gateway Lists и Gateway Rules.
```

Сырой JSON можно показывать только в режиме `--debug`.

### 4.4. Всегда показывать итог

После каждой команды должен быть краткий результат:

```text
Готово.
Списков создано: 13
Правило создано: да
Доменов добавлено: 12 384
State-файл обновлён: .cf-zt-oisd-state.json
```

### 4.5. Минимум терминологии без объяснения

Если используется термин `chunk`, `rule`, `list`, `state file`, рядом должно быть понятное описание.

Пример:

```text
Chunk 001 — часть списка OISD, максимум 1000 доменов.
```

---

## 5. Команды интерфейса

CLI должен поддерживать следующие основные команды:

```text
cf-zt-oisd-sync setup
cf-zt-oisd-sync check
cf-zt-oisd-sync dry-run
cf-zt-oisd-sync init
cf-zt-oisd-sync update
cf-zt-oisd-sync status
cf-zt-oisd-sync delete
cf-zt-oisd-sync doctor
```

Команды должны иметь короткие описания в `--help`.

---

## 6. Команда `setup`

### 6.1. Назначение

Помочь пользователю создать `.env`-файл.

Команда не должна создавать объекты в Cloudflare.

### 6.2. Поведение

При запуске:

```text
cf-zt-oisd-sync setup
```

программа должна спросить:

```text
Введите Cloudflare Account ID:
Введите Cloudflare API Token:
Источник OISD small [https://small.oisd.nl]:
Префикс списков [oisd-small-auto]:
Название правила [OISD Small Auto Block]:
Размер части списка [1000]:
```

API token при вводе должен быть скрыт.

После заполнения программа создаёт `.env`.

### 6.3. Итоговый вывод

```text
Файл .env создан.
Следующий шаг: проверьте подключение командой:
cf-zt-oisd-sync check
```

---

## 7. Команда `check`

### 7.1. Назначение

Проверить, что конфигурация корректна и Cloudflare API доступен.

### 7.2. Проверки

Команда должна проверить:

1. найден ли `.env`;
2. заполнен ли `CLOUDFLARE_API_TOKEN`;
3. заполнен ли `CLOUDFLARE_ACCOUNT_ID`;
4. доступен ли Cloudflare API;
5. есть ли права на Gateway Lists;
6. есть ли права на Gateway Rules;
7. доступен ли источник OISD;
8. можно ли распарсить хотя бы 100 доменов из OISD small.

### 7.3. Вывод

Вывод должен быть в виде списка проверок:

```text
Проверка конфигурации

[OK] .env найден
[OK] Cloudflare Account ID найден
[OK] API token найден
[OK] Cloudflare API доступен
[OK] Доступ к Gateway Lists есть
[OK] Доступ к Gateway Rules есть
[OK] OISD small доступен
[OK] Доменов после обработки: 12 384
```

Если есть ошибка:

```text
[ERROR] Cloudflare API вернул 403 Forbidden

Возможная причина:
API token не имеет нужных прав.

Что сделать:
1. Откройте Cloudflare dashboard.
2. Проверьте permissions токена.
3. Добавьте права для Zero Trust Gateway Lists и Gateway Rules.
```

---

## 8. Команда `dry-run`

### 8.1. Назначение

Показать, что программа собирается сделать, без изменений в Cloudflare.

### 8.2. Поведение

Команда должна:

1. скачать OISD small;
2. нормализовать домены;
3. разбить список на части по 1000 записей;
4. проверить существующие списки и правила в Cloudflare;
5. показать план действий.

### 8.3. Вывод

Пример:

```text
Предварительный просмотр изменений

Источник: https://small.oisd.nl
Доменов загружено: 12 640
Доменов после очистки: 12 384
Размер части: 1000
Нужно списков Cloudflare: 13

План действий:
+ Создать списков: 13
~ Обновить списков: 0
- Удалить лишних списков: 0
+ Создать DNS rule: да

Изменения НЕ будут применены.
Чтобы применить:
cf-zt-oisd-sync init
```

### 8.4. Требование

`dry-run` должен быть безопасным и не должен вызывать `POST`, `PUT`, `PATCH`, `DELETE` в Cloudflare API.

---

## 9. Команда `init`

### 9.1. Назначение

Создать списки и DNS rule в Cloudflare.

### 9.2. Перед созданием

Если программа видит, что state-файл отсутствует и объектов с таким префиксом нет, она показывает план и спрашивает подтверждение:

```text
Будет создано:
13 Cloudflare DOMAIN lists
1 DNS Gateway blocking rule

Продолжить? [y/N]:
```

Флаг `--yes` должен пропускать подтверждение.

### 9.3. Прогресс

Во время создания показывать прогресс:

```text
Создание списков Cloudflare
[#######-------------] 5/13
```

После создания списков:

```text
Создание DNS Gateway rule...
```

### 9.4. Итог

```text
Готово.
Создано списков: 13
Создано правил: 1
Доменов добавлено: 12 384
State-файл: .cf-zt-oisd-state.json

Следующее обновление можно выполнить командой:
cf-zt-oisd-sync update
```

---

## 10. Команда `update`

### 10.1. Назначение

Обновить уже созданные списки по свежему OISD small.

### 10.2. Поведение

Команда должна:

1. скачать свежий OISD small;
2. посчитать hash списка;
3. сравнить с текущим state;
4. определить, какие chunk-списки нужно обновить;
5. создать недостающие списки, если их стало больше;
6. удалить лишние списки, если их стало меньше;
7. обновить DNS rule, если изменились ID списков;
8. обновить state-файл.

### 10.3. Вывод, если изменений нет

```text
OISD small не изменился.
Доменов: 12 384
Списков Cloudflare: 13
Обновление не требуется.
```

### 10.4. Вывод, если изменения есть

```text
Найдены изменения OISD small

Было доменов: 12 384
Стало доменов: 12 512
Было списков: 13
Стало списков: 13

План:
~ Обновить списков: 4
+ Создать списков: 0
- Удалить списков: 0
~ Обновить DNS rule: нет

Продолжить? [y/N]:
```

Для автоматизации должен быть флаг:

```text
--yes
```

---

## 11. Команда `status`

### 11.1. Назначение

Показать текущее состояние синхронизации.

### 11.2. Вывод

Рекомендуемый формат:

```text
Статус cf-zt-oisd-sync

Конфигурация:
Источник OISD: https://small.oisd.nl
Префикс списков: oisd-small-auto
Название правила: OISD Small Auto Block
Размер части: 1000

Локальное состояние:
State-файл: найден
Последняя синхронизация: 2026-05-08 12:00 UTC
Доменов при последней синхронизации: 12 384
Списков в state: 13

Cloudflare:
Найдено managed lists: 13
DNS rule найдено: да
DNS rule включено: да

Состояние:
[OK] Локальное состояние совпадает с Cloudflare
```

Если есть расхождение:

```text
[WARNING] В Cloudflare найдено 12 списков, а в state-файле указано 13.

Рекомендуемое действие:
cf-zt-oisd-sync doctor
```

---

## 12. Команда `delete`

### 12.1. Назначение

Удалить все объекты, созданные программой.

### 12.2. Безопасное подтверждение

Удаление должно быть максимально защищённым.

Перед удалением показать:

```text
Будут удалены:
DNS rules: 1
Cloudflare lists: 13

Префикс списков: oisd-small-auto
Название правила: OISD Small Auto Block

Это действие нельзя отменить.
Чтобы подтвердить, введите DELETE:
```

Если пользователь ввёл не `DELETE`, операция отменяется.

### 12.3. Флаг `--yes`

Для автоматического удаления:

```text
cf-zt-oisd-sync delete --yes
```

Даже с `--yes` программа должна удалять только managed-объекты:

- имя начинается с `LIST_PREFIX`;
- или объект есть в state-файле;
- description содержит `Managed by cf-zt-oisd-sync`.

### 12.4. Итог

```text
Удаление завершено.
Удалено DNS rules: 1
Удалено Cloudflare lists: 13
State-файл удалён: да
```

---

## 13. Команда `doctor`

### 13.1. Назначение

Диагностика проблем и восстановление после частичных ошибок.

### 13.2. Что проверять

Команда должна проверять:

1. есть ли state-файл;
2. валиден ли JSON state-файла;
3. существуют ли списки из state-файла в Cloudflare;
4. есть ли лишние списки с тем же префиксом;
5. существует ли DNS rule;
6. включено ли правило;
7. содержит ли правило актуальные list IDs;
8. совпадает ли количество chunks;
9. есть ли дубликаты managed lists.

### 13.3. Вывод

```text
Диагностика

[OK] State-файл валиден
[OK] Все списки из state найдены в Cloudflare
[WARNING] Найден один лишний список с префиксом oisd-small-auto
[OK] DNS rule найдено
[OK] DNS rule включено
[ERROR] DNS rule не содержит list ID oisd-small-auto-013

Рекомендация:
cf-zt-oisd-sync update --repair
```

## 14. Визуальный стиль CLI

### 14.1. Цвета

Если используется библиотека `rich`, применять цвета аккуратно:

- зелёный — успех;
- жёлтый — предупреждение;
- красный — ошибка;
- голубой/синий — информационные блоки;
- серый — технические детали.

Нельзя полагаться только на цвет. Рядом должны быть текстовые маркеры:

```text
[OK]
[WARNING]
[ERROR]
[INFO]
```

### 14.2. Таблицы

Для списков, статусов и планов использовать таблицы.

Пример:

```text
Cloudflare lists

Name                  Items   Status
-------------------   -----   ------
oisd-small-auto-001   1000    OK
oisd-small-auto-002   1000    OK
oisd-small-auto-013    384    OK
```

### 14.3. Прогресс-бары

Использовать прогресс-бары для долгих операций:

- скачивание OISD;
- создание списков;
- обновление списков;
- удаление списков.

### 14.4. Краткость

По умолчанию вывод должен быть понятным, но не чрезмерно длинным.

Подробный технический вывод — только через:

```text
--verbose
```

Сырой debug-вывод — только через:

```text
--debug
```

---

## 15. Machine-readable режим

Для автоматизации добавить флаг:

```text
--json
```

В этом режиме программа должна выводить результат в JSON.

Пример:

```json
{
  "success": true,
  "command": "status",
  "domain_count": 12384,
  "chunk_count": 13,
  "cloudflare_lists_found": 13,
  "rule_found": true,
  "state_ok": true
}
```

В режиме `--json` не использовать цветной вывод, таблицы и прогресс-бары.

---

## 16. Exit-коды

CLI должен возвращать стабильные exit-коды:

```text
0  — успех
1  — общая ошибка
2  — ошибка конфигурации
3  — ошибка Cloudflare API
4  — ошибка загрузки OISD
5  — ошибка валидации данных
6  — опасное действие отменено пользователем
7  — найдено расхождение state и Cloudflare
```

---

## 17. Тексты ошибок

Ошибки должны быть написаны обычным человеческим языком.

Плохой пример:

```text
HTTP 403: forbidden
```

Хороший пример:

```text
[ERROR] Cloudflare отказал в доступе: 403 Forbidden.

Скорее всего, API token не имеет нужных прав.
Проверьте permissions токена для Zero Trust Gateway Lists и Gateway Rules.
```

Плохой пример:

```text
JSONDecodeError at line 1 column 1
```

Хороший пример:

```text
[ERROR] State-файл повреждён или не является корректным JSON.

Файл: .cf-zt-oisd-state.json
Что можно сделать:
1. Запустите cf-zt-oisd-sync doctor
2. Если восстановление не нужно, удалите state-файл и выполните init заново
```

---

## 18. Первый запуск пользователя

Идеальный сценарий первого запуска:

```text
cf-zt-oisd-sync setup
cf-zt-oisd-sync check
cf-zt-oisd-sync dry-run
cf-zt-oisd-sync init
cf-zt-oisd-sync status
```

В README это должно быть показано как основной путь.

---

## 19. Поведение при повторном `init`

Если пользователь повторно запускает `init`, а списки уже есть, программа не должна молча создавать дубли.

Нужно показать:

```text
[WARNING] Похоже, программа уже была инициализирована.

Найдено managed lists: 13
Найдено DNS rule: да
State-файл: найден

Что можно сделать:
1. Обновить существующую установку: cf-zt-oisd-sync update
2. Посмотреть статус: cf-zt-oisd-sync status
3. Удалить и создать заново: cf-zt-oisd-sync delete, затем init
```

---

## 20. Поведение при отсутствии state-файла

Если state-файл отсутствует, но в Cloudflare есть списки с нужным префиксом, программа должна предупредить:

```text
[WARNING] State-файл не найден, но в Cloudflare есть списки с префиксом oisd-small-auto.

Это может быть старая установка или ручные объекты.
Для диагностики выполните:
cf-zt-oisd-sync doctor
```

Не удалять и не изменять такие объекты без подтверждения.

## 21. Требования к README с точки зрения UX

README должен быть написан так, чтобы пользователь мог пройти настройку без знания кода.

Обязательные разделы:

1. Что делает программа;
2. Что она создаёт в Cloudflare;
3. Какие права нужны API token;
4. Как создать `.env`;
5. Первый запуск;
6. Как проверить, что всё работает;
7. Как обновлять;
8. Как удалить;
9. Как запустить по расписанию;
10. Частые ошибки.

В README нужно явно написать:

```text
Программа не изменяет существующие Cloudflare policies, если они не были созданы этой программой.
```

## 22. Требования к help-экрану

Команда:

```text
cf-zt-oisd-sync --help
```

должна показывать:

```text
cf-zt-oisd-sync — синхронизация OISD small с Cloudflare Zero Trust Gateway

Команды:
  setup     Создать .env конфигурацию
  check     Проверить Cloudflare API и источник OISD
  dry-run   Показать план без изменений
  init      Создать списки и DNS rule
  update    Обновить списки
  status    Показать состояние
  delete    Удалить созданные программой объекты
  doctor    Диагностика и восстановление

Пример первого запуска:
  cf-zt-oisd-sync setup
  cf-zt-oisd-sync check
  cf-zt-oisd-sync dry-run
  cf-zt-oisd-sync init
```

## 23. Требования к UX автоматизации

Для автоматического запуска должны быть удобны команды:

```text
cf-zt-oisd-sync update --yes
cf-zt-oisd-sync update --yes --json
cf-zt-oisd-sync status --json
```

Автоматический режим не должен задавать интерактивных вопросов.

Если нужна интерактивная операция, но пользователь запустил программу в CI без `--yes`, нужно завершиться с понятной ошибкой:

```text
[ERROR] Требуется подтверждение, но программа запущена в неинтерактивном режиме.
Добавьте флаг --yes, если хотите выполнить действие автоматически.
```

## 24. Требования к локализации

На первом этапе интерфейс может быть на русском языке.

Но архитектурно желательно вынести тексты сообщений так, чтобы позже можно было добавить украинский и английский языки.

Минимальное требование:

```text
LANGUAGE=ru
```

в `.env` или настройках.

В будущем:

```text
LANGUAGE=uk
LANGUAGE=en
```

## 25. Необязательный графический интерфейс в будущем

Основной интерфейс — CLI.

Но архитектура должна позволять в будущем добавить GUI или web-интерфейс.

Поэтому бизнес-логика не должна быть смешана с выводом в терминал.

Разделить:

```text
core sync logic
Cloudflare API client
OISD parser
state manager
CLI presentation layer
```

## 26. Критерии готовности интерфейса

Интерфейс считается готовым, если:

```text
первый запуск можно пройти по командам setup → check → dry-run → init
все опасные действия требуют подтверждения
есть dry-run для просмотра изменений
есть понятный status
есть doctor для диагностики
ошибки написаны человеческим языком
есть --json для автоматизации
есть --verbose и --debug
есть стабильные exit-коды
README объясняет процесс без знания кода
```

## 27. Главный UX-принцип

Пользователь не должен бояться запускать программу.

Перед каждым изменением программа должна ясно ответить на три вопроса:

1. Что будет сделано?
2. Какие объекты Cloudflare будут затронуты?
3. Как отменить или безопасно проверить действие заранее?
